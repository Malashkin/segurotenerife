# RAG-агент консультанта подбора

Публичный чат отвечает на свободные вопросы по нашим страховым услугам:
развёрнуто, на языке пользователя (ru/uk/en/es), **бренд-нейтрально**.

## Поток
1. **Корпус** — `knowledge-base/asisa/services.json`: бренд-НЕЙТРАЛЬНЫЙ набор
   сервис-доков (типы покрытия, условия, лимиты, expat-заметки) на каноничном
   испанском + мультиязычные `keywords`. Здесь НЕТ названия страховщика и
   брендовых названий продуктов. Источник фактов — `asisa.es` (нейтрализовано).
2. **Ретривал** — `backend/src/knowledge.rs`: `retrieve(query, intent, k)` =
   буст по интенту чата (med|dental|pet…) + лексический скоринг по `keywords`/
   полям. Кросс-язычно (ru/uk/en-запрос → es-док) без эмбеддингов: корпус мал,
   интент известен. pgvector — возможный апгрейд при росте контента.
3. **Генерация** — `backend/src/routes/chat.rs` (`POST /api/chat`): кэшируемый
   системный блок (правила + индекс всех сервисов) + per-query блок
   ретривнутых фактов; Claude отвечает развёрнуто на языке пользователя
   (`max_tokens=1400`, prompt caching на системном блоке).
4. **Бренд-гейт** — `strip_brand()`: пост-проверка ответа вырезает случайные
   упоминания страховщика (`asisa`/`ocaso`) и логирует утечку. Политику не
   доверяем модели — это детерминированная проверка кодом.

## Жёсткие правила (политика нейтрального бренда)
- НИКОГДА не называть страховую компанию и брендовые названия продуктов —
  говорим о ТИПАХ покрытия; конкретику называет менеджер.
- ЦЕНЫ не публикуются → не называть сумм; считает менеджер.
- Отвечать только по корпусу; чего нет — «уточнит менеджер».
Защита многоуровневая: (а) корпус бренд-нейтрален → модель физически не видит
бренд; (б) правило в системном промпте; (в) `strip_brand`-гейт на выходе.

## Контракт
`POST /api/chat` → `{ question, lang?, intent? }` → `{ answer }`.
Фронт: `askQuestion(question, lang, intent?)` (`@shared/api`); `ChatWidget`
передаёт активный интент карточки. Без `ANTHROPIC_API_KEY`/корпуса → 503,
фронт молча откатывается к гайдовому чату.

## Как расширять
- **Контент:** добавить/обогатить доки в `services.json` (бренд-нейтрально!),
  проставить `intents` и мультиязычные `keywords`. Перевод на язык клиента —
  на этапе ответа (LLM), хранить 4 языка не нужно.
- **Линии:** маппинг интент→сервисы задаётся полем `intents` в доке.

## Тесты
- `backend`: `cargo test knowledge::` — ретривал (интент, кросс-язык, фолбэк) и
  `strip_brand` (7 кейсов).
- `frontend e2e`: свободный вопрос рендерится в чате (стаб `/api/chat`).
- **Живой тест агента** требует `ANTHROPIC_API_KEY` + БД: поднять backend и
  дёрнуть `POST /api/chat` (стаб-ключа недостаточно — пойдёт реальный вызов Claude).

## Стоимость
Каждый свободный вопрос = 1 вызов Claude. Модель по умолчанию —
**`claude-haiku-4-5`** (дёшево; для более «умных» ответов — `claude-sonnet-4-6`/
`claude-opus-4-8` через `ANTHROPIC_MODEL`). Системный блок кэшируется (prompt
caching ≈0.1× за чтение).

## Наблюдаемость
Каждый ответ трассируется в Langfuse (вопрос/ответ/токены/латентность, группировка
по сессии) — см. [../monitoring/observability.md](../monitoring/observability.md).

# Knowledge base — Seguro Tenerife

База знаний, на основе которой чат-бот консультирует клиентов **до** передачи
менеджеру. Источник продуктовых данных — публичный сайт **ASISA** (asisa.es).

## Зачем
Бот должен отвечать на вопросы посетителей («что покрывает полис для ВНЖ?»,
«есть ли стоматология для семьи?») фактами из проверенной базы, а не выдумывать.
Архитектура — RAG: на вопрос ищем релевантные продукты в базе → передаём их как
контекст в Claude API → бот отвечает на языке клиента (ru/uk/en/es) и при интересе
передаёт лид менеджеру.

## Структура
```
knowledge-base/
├── README.md                  ← этот файл
└── asisa/
    ├── schema.json            ← JSON Schema одной записи продукта
    ├── catalog.json           ← все продукты ASISA (источник истины базы)
    ├── index.json             ← сводка по линиям (генерится из catalog.json)
    ├── residency-visa.md      ← highlight: продукты для ВНЖ/визы (ключ для экспатов)
    └── SOURCES.md             ← список источников, дата, дисклеймеры
```

## Принципы данных (важно)
- **Цены.** ASISA почти не публикует тарифы. Где не опубликовано — `pricing_notes: null`.
  НЕ выдумывать цены: считать в тарификаторе asisa.es или по телефону 900 123 470.
- **Язык.** `coverage`/`exclusions` хранятся на испанском (как на сайте) —
  это первоисточник. `summary_ru`/`summary_en` — короткие крючки. Детальный
  перевод под язык клиента делает LLM на этапе ответа.
- **Актуальность.** ASISA периодически переименовывает модальности salud. Перед
  использованием с клиентом сверяться с `official_url`. Дата сбора — в `index.json`.
- **Чего нет.** ASISA НЕ продаёт страхование жилья (Hogar). Decesos (похоронное)
  андеррайтит OCASO, ASISA — дистрибьютор.

## Как обновлять
1. Перепроверить продукты по `official_url` (или заново прогнать разведку).
2. Обновить `catalog.json` (поля по `schema.json`).
3. Перегенерировать `index.json` (см. скрипт в истории/CI).
4. Обновить дату `retrieved`.

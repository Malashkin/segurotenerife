### rate_limit.rs (защита от DDoS/cost-DoS) — 8 тестов
Лимит до порога → блок; per-IP изоляция; **сброс окна** (детерминированно через
инъекцию `now`); **граница окна эксклюзивна** (ровно 60с ещё лимит); **эвикция**
протухших IP; `client_ip` из XFF/CF-заголовков при доверии прокси и игнор без;
ответ лимита = **429**.

> `allow_at(ip, now)` с инъектируемыми часами (детерминизм окна/эвикции без `sleep`).

### Выжившие мутанты — обоснование (равноценные / низкий риск / integration-level)
- `knowledge.rs:121 > → >=` (граница `any_positive`) — **эквивалентный**: счёты
  дискретны, наблюдаемый исход (фолбэк vs фильтр) не меняется.
- `knowledge.rs:137 || → &&` (keyword `==` vs `contains`) — отличие лишь в
  подстрочном матче keyword; его страхует haystack-сигнал (+2); топ-результат не
  меняется. Низкий риск.
- `knowledge.rs:141 += → -=/*=` — это слабый haystack-сигнал (+2); поведение
  доминируется протестированными keyword (+10) и intent (+100). Низкий риск.
- `knowledge.rs:216 strip_brand + → *` — **TIMEOUT**: мутация даёт выход за
  границы среза (panic/зависание) → фактически детектится, не «тихий» выживший.
- `rate_limit.rs:63 > → >=` (триггер свипа эвикции) — **эквивалентный** для
  корректности: свип — best-effort GC, момент срабатывания не влияет на allow/deny.
- `rate_limit.rs:129/145 middleware → Default::default()` — **integration-level**:
  middleware — тонкая склейка из юнит-покрытых `client_ip` + `limiter.allow` +
  `too_many`/`next`. Рекомендация: интеграционный тест роута через
  `tower::ServiceExt::oneshot` (требует поднятия router+AppState с БД).
